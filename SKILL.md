# Sky Media Article Creator Skill

## 概览（Overview）

用于从主题或大纲创作中文公众号 / 自媒体文章，默认采用“故事 + 观点”的写作风格。最终输出为适合直接粘贴到公众号、头条号等平台的**纯文本内容**，不使用 Markdown 语法（不包含 #、*、``` 等标记），必要的层次结构通过自然分段和小标题句子表达。

核心能力：
- 帮助明确切入角度与目标读者
- 提供多组标题候选与结构化大纲
- 在用户确认大纲后，生成成稿，并以简洁要点总结收尾

## 使用总则

当请求满足以下任一特征时，触发并使用本技能：
- 用户说“以……为核心/主题/内容，写一篇文章”
- 用户说“写一篇关于……的文章”
- 用户说“创建自媒体文章 / 公众号文章”等类似表述

触发后，优先做两件事：
1. 澄清写作关键信息：主题/主线、是否已有大纲、期望读者画像（如果用户未主动说明）。
2. 统一写作参数：文风、字数范围、是否需要多组备选标题。

在信息不完整时，不直接写文，而是先发问补齐。

流程闸门（强约束）：
- 默认必须执行“两步法”：先给标题与大纲，用户确认后再写全文。
- 只有当用户明确表示“直接一稿”“不用先看标题大纲”时，才允许进入快速模式。
- 若用户未明确选择快速模式，禁止直接输出完整正文。

默认语气与营销边界：
- 语气：专业但不端着、真诚、不夸张，适度幽默即可，避免过度煽情。
- 对产品/APP 的介绍：以真实使用场景和价值为主，不写成生硬植入或纯广告语堆砌。
- `soft-intro`（弱宣传）模式：当用户强调“不要像广告/宣传文”时自动启用。
  - 主体叙事比例：至少 60% 篇幅用于真实场景、取舍与反思；功能点介绍不超过 40%。
  - 限制词：避免“最强/颠覆/革命性/必备神器”等夸张表达。
  - CTA 限制：全文最多 1 处轻量行动句，且不能放在首段。

快速模式（可选）：
- 如用户明确表示“直接一稿写完，不用中间确认标题和大纲”“直接生成完整文章”等，可进入快速模式。
- 快速模式下，仍需在正文前简要给出【推荐标题 + 文章大纲】，随后紧接着输出全文。

## 交互步骤一：确认需求与写作参数

1. 判断用户当前给了什么：
   - 仅有模糊主题（如“写一篇关于 AI 的文章”）
   - 主题 + 若干要点
   - 明确大纲/结构
   - 或一篇已有草稿（长文本，需改写/升级为公众号文）

2. 若主题或主线不清晰时，引导用户补充（示例引导）：
   - “你更想强调的是【工具介绍 / 使用体验 / 背后故事 / 方法论】中的哪一部分？”
   - “这篇文章主要是给谁看的？（例如：学生、职场新人、某个岗位的人）”

3. 主动询问并记录以下参数：
   - 文风：如用户未指定，默认采用“故事+观点”的公众号常见口语化文风，兼顾可读性和观点清晰度；也可以在下列预设中让用户选择：
     - 感性故事型：情绪和细节更多一些，更像讲故事，适合个人经历、成长类主题。
     - 干货理性型：结构更紧凑、信息密度高，少量故事铺垫，适合教程、经验总结、功能介绍。
     - 职场专业型：语气稍微正式一点，但不过于官话，适合面对职场读者或行业人士。
     - 轻松吐槽型：语气更随意，适度调侃和自嘲，适合用户心声、踩坑分享、日常观察。
   - 当用户没有明确选择时，默认使用“感性故事型 + 干货理性型”的混合风格：故事打头，观点收束。
   - 字数：如用户未指定，目标控制在 1500–2000 字；用户若给出区间，以用户为准。
   - 标题数量：默认提供 3–5 个备选标题。
   - 封面与插图需求：
     - 是否需要生成封面图？如果需要，让用户指定封面图比例（如 16:9、3:2、1:1 等）。
     - 是否需要正文插图？如果需要，可询问大致希望哪些部分有图（例如：方法步骤、关键对比、复盘时间轴等）。

4. 确认完成后，总结当前理解，用 2–4 句话复述需求要点再进入下一步。

5. 如用户给出官网/文档链接，先执行“事实抽取”再写：
   - 抽取 3 类信息：功能名、每个功能一句价值、可验证的边界条件（如本地处理、加密、支持的输入来源等）。
   - 对不确定信息标记为“待确认”，正文中不用猜测性措辞补全。
   - 功能介绍段严格来自已抽取事实，避免杜撰参数、价格、兼容性。

## 交互步骤二：输出标题与大纲（先审稿，再写文）

在写正文之前，先输出「标题 + 读者与价值点说明 + 大纲」，供用户确认或修改。

1. 标题设计：
   - 给出 3–5 个候选标题，整体风格偏稳重、专业、真诚，符合“故事+观点”的公众号调性。
   - 标题尽量兼顾：可读性 + 信息量 + 适度吸引力，避免夸张标题党和过度悬疑式表达。
   - 标记一个“推荐标题”，并说明推荐理由（1–2 句），理由中可点出“为什么更符合这篇故事和观点的气质”。

2. 目标读者与价值点说明：
   - 用 1–3 句话明确：  
     - 主要面向的读者群体（如：在互联网公司工作的产品新人 / 想入门 AI 的大学生等）  
     - 读者读完这篇文章能获得的核心价值（如：获得某类工具的使用思路、看到真实开发故事、学到可复制的方法等）。

3. 文章结构与大纲（采用固定结构的变体）：
   - 推荐基础结构：
     1. 开篇故事/场景：用一个具体人或具体场景引入问题（故事感、画面感）。
     2. 背景与问题铺陈：点出读者的共鸣点和典型困境。
     3. 关键观点或方法拆解：2–4 个小节，分别展开核心观点/步骤/案例。
     4. 延伸思考或实践建议：给出可操作的建议或更高一层的思考。
     5. 结尾简洁总结：用简短文字提炼要点，避免冗长鸡汤。
   - 使用编号和小标题，输出清晰的大纲，每一小节附 1–2 句说明该节要讲什么。

4. 输出格式建议：
   - **标题候选**
   - **推荐标题与推荐理由**
   - **目标读者与阅读价值**
   - **文章大纲**

5. 明确提示用户：
  - 请用户选择或调整标题
  - 请用户对大纲提出修改意见或补充要点
  - 只有在用户确认后，才进入全文写作步骤
  - 若用户未确认，则继续迭代标题与大纲，不进入正文

## 交互步骤三：生成全文（故事 + 观点）

在用户确认标题和大纲后，再生成完整文章。

1. 篇幅与节奏控制：
   - 如无特别说明，整体字数控制在约 1500–2000 字。
   - 避免长篇无分段，确保每个小节有清晰的段落划分。
   - 若用户给了字数区间，按区间执行；若超上限 10% 以上，必须进行二次压缩后再输出。

2. 开头写法：
   - 采用故事/场景/个人经历式开头，用 2–5 个自然段形成画面感。
   - 当用户提供了主题、大纲或其他内容素材时，优先从这些素材中发散、提炼出能打动读者的小故事或具体场景作为开篇，例如：
     - 当主题是介绍/宣传一款工具类 APP 时，可以从“作者如何因为某个痛点而开始开发这款 APP”的经历讲起，再自然过渡到功能介绍与使用体验。
   - 在开头中自然埋入文章核心问题或冲突，让读者知道“为什么要读下去”。

3. 中间展开：
   - 严格按照已确认的大纲组织内容，不随意新增大段结构性内容。
   - 每个小节兼顾：故事细节 / 具体案例 + 清晰观点或方法总结。
   - 适度使用小标题、列表，提升可扫读性。

4. 文风要求：
   - 总体口语化、真诚、具体，尽量像一个真实的人在和读者聊天，而不是正式报告，避免空泛口号和堆砌鸡汤。
   - 句式长短有变化，避免整篇都是“首先、其次、再次”这类模板化结构，适度使用第一人称“我”来表达真实感受。
   - 观点要落地：多用“具体场景 + 具体行动 + 具体结果”的表达，让读者能在脑海里看到画面。
   - 当用户选择不同风格预设时，应做以下微调：
     - 感性故事型：多给一点细节和情绪描写，但仍避免夸张和无意义的煽情。
     - 干货理性型：减少感性铺垫，快速进入结构化观点和步骤，但保留少量具体例子。
     - 职场专业型：用词略偏正式，注意逻辑严谨和表达清晰，但避免过度官话和公文式表达。
     - 轻松吐槽型：适度加入调侃、自嘲和对话式句子，但不要滑向无意义的抱怨，仍需有观点和收束。

5. 结尾写法（必须遵守）：
   - 用简洁方式总结全文要点，可用 3–5 条短句或一小段紧凑文字。
   - 避免冗长的重复回顾和泛泛而谈的鸡汤式收尾。
   - 若自然合适，可以带一句轻量级行动号召（如“如果有类似经历，可以留言聊聊”等），但不强制。

6. 输出前自检（自查清单）：
   - 检查是否有明显重复、空洞或纯堆砌段落，必要时适度压缩。
   - 检查内容是否严格围绕已确认的大纲，有无跑题的大段扩写。
   - 检查结尾是否确实以简洁要点总结为主，而不是再次长篇讲故事或泛泛鸡汤。

## 处理已有草稿的情况（可选）

当用户提供一篇已有文章或长篇草稿时：
- 优先识别用户意图：是要“改写成更适合公众号的版本”，还是“精简/扩写”，或“统一为故事+观点风格”。
- 保留原文核心信息与事实，但可重构结构、重写开头与结尾。
- 同样给出：目标读者与价值点说明 + 新结构大纲，再给出改写后的全文。

## 个人语气与风格沉淀（references）

为了让生成的文章越来越贴近你的个人风格，本技能可以配合 `references/personal_tone_examples.md` 使用：

- 当该文件存在且不为空时，在写文章或重写段落前，可优先简要阅读其中内容，提炼你的常见语气特征，例如：
  - 句子长短节奏（是否常用短句、是否习惯中间有停顿）
  - 常用口头表达（如“老实说”“有点”“说实话”等）
  - 叙述视角（更多用“我”，还是更多用“你/我们”）
- 写作时尽量在以下方面对齐该文件中的示例：
  - 用词风格：尽量复用你喜欢的表达方式，而不是统一的官方腔。
  - 情绪强度：是偏克制、偏热情，还是带一点自嘲、幽默。
  - 结构节奏：例如一段故事 + 一句简短感受 + 一小段观点，这种组合可以在新文章里重复使用。
- 当你对某篇文章的语气特别满意时，可以手动把其中的一两段粘贴到该 references 文件中，配上简单说明（例如“我想要的功能介绍风格大概就是这样”），作为后续文章的风格基准。

注意：为了节省上下文，本技能在使用该 references 文件时应做到“点到为止”，只加载和参考少量最典型的段落，不需要每次全部读完。

## 常见文章类型模板

可根据用户主题与受众，从下列模板中择近似类型作为参考结构。结构可以微调，但应保持“故事 + 观点”的基本思路。

1. 工具 / APP 介绍类
   - 适用：产品介绍、上线公告、使用经验分享等。
   - 结构示例：
     1. 起因故事：我是如何遇到某个痛点，或为什么会想到做/用这款工具。
     2. 痛点与现状：过去是怎样低效/麻烦的，读者可能也有同样困扰。
     3. 工具诞生/选择历程：开发过程/选择过程中的关键节点和取舍。
     4. 功能亮点与使用场景：用具体场景串起来讲 2–4 个核心功能。
     5. 使用建议与适用人群：谁最适合用、怎么用收益最大。
     6. 简洁总结：重新点题 + 强调核心价值。

2. AI 知识科普 / 应用案例类
   - 适用：介绍某个 AI 概念、模型、应用场景等。
   - 结构示例：
     1. 生活/工作中的一个具体场景：读者熟悉的画面，引出“其实这里已经在用 AI / 可以用 AI”。
     2. 典型误解或疑问：普通人对该技术的常见误区。
     3. 核心概念拆解：用类比/比喻讲清楚关键概念（尽量避免纯术语堆砌）。
     4. 应用案例：具体说明 2–4 个真实或贴近现实的应用场景。
     5. 风险与边界：简单提醒可预期的风险、局限。
     6. 实用建议：普通人/某类读者可以怎么开始尝试。
     7. 简洁总结：回扣开头场景，强化“理解 + 可行动”。

3. 产品经理经验 / 能力拆解类
   - 适用：职业经验分享、能力模型拆解、职业建议等。
   - 结构示例：
     1. 真实工作片段/冲突：讲一个典型“踩坑”或冲突场景。
     2. 问题本质分析：拆解这背后真正考验的是哪几类能力。
     3. 能力拆解：按 2–4 个能力点展开（如：沟通、推进、判断、抽象等）。
     4. 具体做法/方法论：每个能力点给出 1–2 个具体做法或小技巧。
     5. 个人反思/成长：这件事对自己的改变或启发。
     6. 给读者的建议：不同阶段读者可以如何实践。
     7. 简洁总结：提炼 3–5 个关键提醒。

4. 个人成长复盘 / 年度总结类
   - 适用：年终总结、项目复盘、阶段性成长记录等。
   - 结构示例：
     1. 起始状态回顾：那一年/那个阶段开始时的自己是什么样。
     2. 关键事件串联：挑 3–5 个转折点事件，用故事方式讲清。
     3. 失败与挫折：直面一两个“翻车经历”，重点说真实感受和转折点。
     4. 学到的几件事：总结 3–5 条真切的认知/方法变化。
     5. 还没做到的部分：坦诚不足，而不是只报喜不报忧。
     6. 给读者的启发：把个人经验抽象成可借鉴的思路。
     7. 简洁总结：一句话钉住这一年的核心关键词或主题。

5. 行业趋势解读 / 观察类
   - 适用：对某个行业变化、政策、热点现象的分析。
   - 结构示例：
     1. 事件/现象切入：从一个具体新闻或身边变化讲起。
     2. 表层现象描述：发生了什么？有哪些看得见的变化。
     3. 深层原因分析：从用户、技术、商业、政策等角度拆开看。
     4. 对不同角色的影响：对个人/从业者/公司分别意味着什么。
     5. 未来可能的走向：在不夸张预测的前提下，给出合理推演。
     6. 给读者的建议：如果是读者，会怎么选择/准备。
     7. 简洁总结：用一两句话收束观点。

6. 踩坑总结 / 反面教材类
   - 适用：分享失败项目、错误决策、踩坑经验。
   - 结构示例：
     1. 失败故事：客观讲清“怎么翻车的”，保留细节。
     2. 当时的错误判断：当下是怎么想的，为什么会做出那样的选择。
     3. 后果与影响：产生了哪些结果，对自己/团队/用户的影响。
     4. 事后复盘：现在回头看，真正的问题出在哪里。
     5. 教训与改法：总结 3–5 条“如果重来会怎么做”的具体建议。
     6. 给读者的提醒：帮助读者识别类似情境、提前避坑。
     7. 简洁总结：用简短话语提炼“这次教训的名字”。

7. 活动 / 项目复盘类
   - 适用：运营活动、产品发布、线下沙龙等复盘。
   - 结构示例：
     1. 项目背景：为什么要做这件事，目标是什么。
     2. 关键设计与策略：活动/项目是怎么设计的，有哪些关键策略。
     3. 过程中的关键节点：列出 3–4 个关键节点或转折点。
     4. 结果与数据：给出核心结果，用尽量直观的数据或指标说明。
     5. 成功点与不足：具体说哪里做得好、哪里没做好。
     6. 下次会怎么调整：形成 3–5 条落地的改进要点。
     7. 简洁总结：回扣最初目标，看清真正的收获。

以上模板都是“参考结构”，实际使用时可以根据用户给的素材和主题做微调，但应保持：有故事、有观点、有结构、有总结。

## 生成后审核与去“AI 味”机制

在完成一稿文章后，不直接视为最终答案，而是进行一轮内部审核与润色，再给出给用户。

1. 字数与结构检查：
   - 对照用户给定的目标字数范围（或默认 1500–2000 字），判断当前篇幅是否明显偏离。
   - 如明显超出或不足，应适度压缩或补充关键内容，避免为凑字数而堆砌无信息段落，宁可略短一些但信息密度高、表达自然，也不要为了长度强行扩充模板化段落。
   - 收敛阈值：
     - 超过上限 10%：必须压缩；
     - 低于下限 10%：仅在信息缺失时补充，若素材不足可维持略短；
     - 压缩优先级：先删空泛形容词，再删重复例子，最后压缩背景铺垫。
   - 检查文章是否完整覆盖已确认的大纲各部分，有无遗漏核心小节。

2. 错别字与语句通顺：
   - 在心中“再读一遍”全文，查找明显的错别字、重复词、残句或语序不顺之处，并在输出前进行修改。
   - 优先处理会影响理解的错误，其次处理不影响理解但影响体验的瑕疵。

3. 去除明显“AI 味”的表达：
   - 避免使用过于模板化、常见于 AI 文本中的句式，例如：  
     “在当今飞速发展的时代”“总而言之”“首先，其次，再次”“由此可见”“不难发现”等，如出现，要改写为更自然、日常的口语表达。
   - 减少过度中立、过度客气的语气（例如“我们不难发现”“可见……”等），多用第一视角或读者视角的自然说法。
   - 优先用具体细节和真实场景替代抽象的空话，必要时可以增加一两处贴近主题的小细节（但不能捏造事实信息）。
   - 如果通读全文后感觉语言过于中立、平铺直叙、像说明书或模版文章，应主动重写关键段落，引入更自然的口语表达、停顿和情绪，让读者能感受到“这是一个具体的人在讲自己的故事”。

4. 保持个人感和真实感：
   - 在不改变事实的前提下，适度加入“我当时怎么想的”“我当时的感受是……”这类主观视角，使文章更像真实作者写的，而不是完全中性的说明文。
   - 避免整段都在讲“普遍真理”却没有具体人和事情的段落，如有这类段落，要么补充故事，要么压缩为几句观点。

5. 审核后输出策略：
   - 审核和润色的过程在内部完成，不需要向用户展示中间修改过程。
   - 向用户展示的应是已经过这一轮“查错 + 调整语气 + 去 AI 味”处理后的版本。

## 局部重写与迭代优化能力

当用户对文章整体基本满意，只想修改其中某一部分时，应优先采用“局部重写”而不是整篇重写，以节省时间并保留已有表达。

1. 明确重写范围：
   - 引导用户说清楚要修改的位置，例如：
     - “从‘二、为什么要在我的应用里做这个功能’这一节开始，到下一节前为止”
     - “第一个小标题下面的第二段话”
     - “结尾总结部分”
   - 若用户描述不够具体，可简要复述你的理解，让用户确认后再动笔。

2. 局部重写原则：
   - 只重写被指定的段落或小节，其余内容保持不变（包括标题、其他小节结构和插图占位符）。
   - 重写时仍需遵守本技能的基本要求：
     - 故事 + 观点结构不被破坏
     - 字数整体仍尽量控制在 1500–2000 字范围内
     - 语言口语化、有个人感、无明显 AI 模板味
   - 若局部重写导致整篇篇幅明显增加，应适度压缩其他部分的冗余段落，保持整体长度平衡。

3. 针对性改进的典型方向：
   - 加强故事感：当用户觉得“这一段太平淡”，可以增加具体场景、对话或细节，让读者有画面感。
   - 提升观点力度：当用户觉得“这里观点太弱”，可以更直接地给出立场、判断，并配上简短解释或例子。
   - 调整语气：当用户觉得“这里太官方/太像广告”，优先改用更自然的口语表达，减少形容词和夸张句式。

4. 输出方式：
   - 默认只返回被重写的那一小段/小节的新版本，并简要说明它将替换原文中的哪一部分，方便用户自行替换。
   - 如用户明确要求“给我整篇更新后的版本”，再在完整文章中替换对应段落后输出全文。

5. 与个人语气 references 的配合：
   - 在局部重写前，可优先参考 `references/personal_tone_examples.md` 中同类型的示例片段，帮助新段落更贴近用户既有风格。
   - 若重写后的段落比原先更贴近用户期望风格，用户可选择将其复制到 references 文件中，作为后续类似场景的写作参考。

## 封面与插图生成工作流

在用户明确希望“同时生成文章 + 封面图 + 插图”时，本技能与脚本配合工作：

1. 询问与确认：
   - 是否需要封面图？如果是，确认封面图的目标比例（例如：16:9、3:2、4:3、1:1 等），记录为后续调用脚本时的默认封面比例。
   - 是否需要正文插图？如果是，结合大纲，判断哪些段落最适合配图（例如：方法步骤、对比说明、时间轴、关键故事节点）。
   - 确认输出文件夹（例如：用户指定某个路径，该路径下会统一保存文章与图片）。
   - 明确默认行为：一旦用户在此阶段选择“需要封面图/插图”，在文章文本定稿后应自动发起封面图与插图的生成流程，而不是等待用户再次单独提出“跑脚本”。

2. 设计图像创意：
   - 在标题与大纲已确认的基础上，先给出：
     - 1–2 个封面图创意（画面内容 + 风格简述），例如：  
       “画面背景是抽象的科技线条或网格，前景用大号中文写着‘图片转 PDF’，同时在画面中加入几张叠放的图片小缩略图与合并成一页的 PDF 图标，整体色调温暖，适合公众号封面。不出现任何人物或人脸。”  
   - 提示词生成规则（核心内容优先）：
     1）先提炼“文章核心内容”一句话（10–20 字），作为图像的唯一主线；  
     2）封面图必须包含“核心主题文字 + 相关元素”，即：除核心主题文字外，再加入 2–4 个与主题强相关的图形/物体元素；  
     3）封面文字不要照搬文章完整标题，应压缩为短文案（2–8 字）或关键词组合（2–4 个词，建议“词A + 词B”）；  
     4）正文插图只对应各小节核心点，不复述整篇文章；每张图只表达 1 个重点动作、重点对比或重点结果；  
     5）若元素与核心内容关联弱（仅好看但无信息），应删除。
     6）默认视觉风格采用“浓郁色调 + 清晰对比”：建议中高饱和、主辅色明确、局部高光点缀；禁止灰暗、低饱和、雾蒙蒙、发闷的整体观感。
   - 封面文案简化示例：
     - 原标题：`我用vibe coding做了一款工具App，也重新理解了AI产品`
     - 封面显示文案：`vibe coding + 工具APP`（或 `AI工具产品`）
   - 封面提示词模板（推荐直接套用）：
     - `中文封面图，主题文字“大号清晰展示：{简化文案（非完整标题）}”。围绕该核心内容加入{元素1}、{元素2}、{元素3}等相关图形/物体元素；背景氛围为{科技/职场/复盘/财经等}；色彩中高饱和、对比清晰、视觉有张力，避免灰暗低饱和；版式简洁、信息层次分明、留白合理、可读性高；不出现人物，不包含人脸，不要人像。`
   - 正文插图提示词模板（推荐直接套用）：
     - `中文信息插图，聚焦“{本节核心点}”，使用{图表/流程/对比/时间轴/界面卡片}表达；仅保留与该核心点直接相关的元素：{元素A}、{元素B}；色彩浓郁、对比明确、画面有活力，避免灰暗低饱和；说明文字短而清晰，不出现人物。`
   - 每个封面创意的提示词应尽量包含以下要素：
     1）主题文字：直接在画面中出现文章的核心关键词（如‘视频转 GIF’、‘自媒体写作’、‘AI 初创公司’等），并用“大号、清晰”的中文标题文字描述；  
     2）背景氛围：根据文章主题选择相匹配的背景氛围，而不是统一模板，例如：  
          - 主题偏 AI / 大模型 / 技术趋势：使用科技线条、芯片、数据流、网络节点、算法图形等，整体科技感强；  
          - 主题偏金融 / 市场 / 生意：使用涨跌曲线、K 线、数字面板、报表、硬币或简化的图表等，整体更偏财经感；  
          - 主题偏职场 / 工具 / 效率：使用电脑、手机、窗口界面、日历、待办清单、白板等办公场景物件，但不出现真实人物；  
          - 主题偏自媒体 / 写作 / 表达：使用纸张、笔记本、键盘、文本气泡、文章段落缩略图等，营造“在写东西”的氛围；  
          - 主题偏成长 / 复盘 / 故事：使用时间轴、路标、脚印、书本、日历页等，传递“过程”和“回看”的感觉；  
       3）象征性图形元素：用 2–3 个与正文核心内容对应的图形或图标来概括文章（例如：关于 AI 初创公司，可以用模型方块、渠道箭头、感叹号图标；关于工具功能，可以用文档、齿轮、时间轴、机器人图标等）；  
       4）禁止人物：在 prompt 中明确写出“不出现人物、不包含人脸、不要人像”，只使用文字、背景和与主题相关的物体/图形元素。
   - 对于正文插图：
     - 在大纲中，对每个需要插图的小节补充一句“建议插图：……”，说明图的类型（流程图/对比图/时间轴/场景插画等）和大致内容。

3. 在文章中嵌入插图标记（HTML 注释）：
  - 在生成最终文章时，不直接插入 Markdown 图片，而是根据上下文，在最合适的段落中插入 IMAGE 标记，供脚本解析：
     - 封面图标记示例（通常放在文章最前面，或单独保存而不嵌入正文）：  
       `<!-- IMAGE: cover | prompt=作者因为自己时间管理混乱而决定做一款时间管理 APP 的场景插画 | aspect=16:9 -->`
     - 正文插图标记示例（放在对应段落前后）：  
       `<!-- IMAGE: inline | name=时间管理前后对比 | prompt=展示使用 APP 前后，工作日程从混乱到清晰的对比插画 | aspect=16:9 -->`
   - IMAGE 标记字段说明：
     - `cover` / `inline`：图片类型（封面/正文插图）。
     - `prompt`：给 AI 生图 API 的提示词，尽量具体、易于理解。
     - `aspect`：期望的图片比例（可选，若缺失则封面使用用户指定的默认比例，插图可由脚本默认处理）。
     - `name`：正文插图在纯文本占位符中的名称（可选）。

4. 调用生图脚本生成图片并更新文章：
   - 当文章定稿并插入了 IMAGE 标记后，调用脚本：
     - 路径：`skills/sky-media-article-creator-skill/scripts/generate_article_assets.py`
     - 示例命令：  
       `python skills/sky-media-article-creator-skill/scripts/generate_article_assets.py --article-file /path/to/article.md --output-dir /path/to/output/folder --cover-aspect 16:9`
  - 脚本行为（见下一节）：
     - 读取文章，解析所有 IMAGE 标记。
     - 调用外部图片生成 API，为每个标记生成一张图片。
     - 将生成的封面图和插图统一保存到 `output-dir` 下，并将正文中的 inline 标记替换为结构化的纯文本插图占位符，方便用户在公众号、头条号等后台手动插入对应图片。例如：  
       `【插图：名称=时间管理前后对比；文件=inline_1.png】`
     - 在同一文件夹下写入更新后的纯文本文章与原始文章副本。
     - 脚本会自动增强提示词：封面图自动强调“核心内容文字 + 相关图形/物体元素”，并自动避免照搬完整标题；正文图自动强调“只围绕段落核心信息”；两者都会附加“色调浓郁、对比清晰、避免灰暗低饱和”的风格约束。
     - 若生图接口偶发失败，脚本会自动重试（指数级短退避），仍失败则抛出错误并保留已完成文件。
   - 调用顺序建议：
     - 若存在封面标记，应先确保封面图生成成功（或至少完成调用），再依次处理正文插图标记，方便用户在自媒体后台先配置封面，再按占位符插入正文图片。

5. 最终产出结构：
   - 一个输出文件夹（例如 `/path/to/output/folder`），包含：
     - `article_with_images.txt`：已经插入插图位置文本标记的纯文本文章，可以直接复制到自媒体后台，再手动插入生成的封面和插图。
     - 原始文章文件副本（例如 `article.md` / `article.txt`）。
     - `cover.png`：封面图（如有）。
     - `inline_1.png`、`inline_2.png`……：正文插图（如有）。

## 脚本：generate_article_assets.py 使用说明

本技能包含一个脚本，用于根据文章中的 IMAGE 标记调用外部 AI 生图 API，生成封面与插图，并将文章与图片统一输出到同一文件夹。

- 脚本路径：  
  `skills/sky-media-article-creator-skill/scripts/generate_article_assets.py`

- 依赖：
  - Python 3
  - `requests` 库（如未安装，需要 `pip install requests`）

- 环境变量（需按你的真实 API 配置）：
  - `IMAGE_API_URL`：生图 API 的 URL。
  - `IMAGE_API_KEY`：API 的访问密钥（如不需要可留空，脚本会跳过 Authorization 头）。

- 基本用法示例：

  ```bash
  python skills/sky-media-article-creator-skill/scripts/generate_article_assets.py \
    --article-file /path/to/article.txt \
    --output-dir /path/to/output/folder \
    --cover-aspect 16:9
  ```

  参数说明：
  - `--article-file`：输入文章文件路径（纯文本更推荐），其中包含 IMAGE 标记。
  - `--output-dir`：输出目录，脚本会在此目录写入更新后的纯文本文章和所有图片。
  - `--cover-aspect`：封面图默认比例（如 `16:9`、`3:2` 等）；可被单个标记中的 `aspect=...` 覆盖。
  - `--updated-article-name`（可选）：更新后文章文件名，默认 `article_with_images.txt`。

- 工作原理概要：
  1. 读取文章内容，查找所有形如 `<!-- IMAGE: ... -->` 的标记。
  2. 解析每个标记的字段（类型、prompt、aspect、name）。
  3. 对于封面标记：
     - 只生成一张封面图，文件名固定为 `cover.png`（多余封面标记会被忽略）。
  4. 对于插图标记：
     - 按顺序生成 `inline_1.png`、`inline_2.png` 等文件。
     - 在文章中用纯文本占位符替换原有标记，格式为：`【插图：名称=<name>；文件=inline_X.png】`，其中 `<name>` 来自 IMAGE 标记中的 `name` 字段（或默认“插图X”），`inline_X.png` 为对应生成的图片文件名。
     - 注意：封面图固定为 `cover.png`，不占用 `inline_X` 序号；正文插图始终从 `inline_1.png` 开始计数。
  5. 将更新后的文章写入 `output-dir/article_with_images.txt`，同时在同一目录保存一份原始文章副本和所有图片。

- 关于 API 适配：
  - 脚本假设生图 API：
    - 接收 JSON 请求体，至少包含 `prompt` 字段，可选 `aspect_ratio` 字段。
    - 直接返回图片二进制内容（即 HTTP 响应体就是图片）。
  - 如果你的 API 返回 JSON（例如图片 URL 或 base64 编码），需要在脚本中修改 `call_image_api` 函数：
    - 调整请求 payload 字段名。
    - 在拿到响应后，解析 JSON，再去下载图片或解码 base64，最终返回图片二进制数据。

配置好 API 后，只要文章中有符合规范的 IMAGE 标记，脚本就能在一个命令里帮你把“文章 + 封面 + 插图”统一生成到同一个文件夹中，方便后续在公众号后台上传使用。
